# ADR 001: UI要素特定における座標依存からの脱却

| 項目 | 内容 |
|------|------|
| 状態 | 承認済み (2026-03-03) |
| 決定者 | Isao Shinohara |
| 関連ファイル | `crawler/lc/crawler.py` — `_extract_title()` |

---

## コンテキスト

Phase 2〜3 において `_extract_title()` は iOS 設定アプリの画面座標を前提とした
固定ピクセル値 (`y=150`, `y=750`, `x=400` 等) で画面タイトルを特定していた。

この実装では以下の問題が顕在化した:

1. **誤判定**: iOS Large Title が画面中央寄り (x≈587px) に表示される場合、
   `x < 400` フィルタが機能せず nav-bar の「< 設定」戻るボタン (x≈95) を
   タイトルとして誤採用した。
2. **移植不可**: スマートフォンゲームの UI やタブレット・異解像度デバイスでは
   ピクセル値が全く異なり、コードの再利用が不可能。
3. **保守コスト**: デバイス追加のたびに座標マジックナンバーの調整が必要になる。

---

## 決定

**座標の絶対値指定を廃止し、画面の視覚的構造と相対的特徴に基づく推論へ移行する。**

具体的には以下の 2 ステップアルゴリズムを採用する:

### Step 1 — Large Title (フォントサイズ優先)

```
候補条件:
  - 画面上部 y < screen_h × 30%  (タイトル表示エリア)
  - bounding-box 高さ ≥ screen_h × 3%  (大きなフォント = タイトル相当)
  - 信頼スコア > 0.8、テキスト長 2〜20 文字、EXCLUDE_TEXTS に非該当

採用: 候補のうち bounding-box 高さが最大のテキスト
```

**根拠:**
iOS Large Title は約 34pt (物理 @3x ≈ 3.3%)、nav-bar の標準テキストは
約 17pt (@3x ≈ 2.5%)。3% という閾値で両者が確実に分離できることを
実測データで確認済み。

| テキスト | 高さ | 比率 | 判定 |
|---------|------|------|------|
| 一般 (Large Title) | 84px | 3.3% | ✅ 採用 |
| 設定 (戻るボタン) | 65px | 2.5% | ❌ 除外 |
| 情報 (nav-bar 中央) | 62px | 2.4% | ❌ 除外 → Step 2 へ |

### Step 2 — Nav-bar Center Title (fallback)

```
候補条件:
  - 画面上部 y < screen_h × 20%  (nav-bar 高さの上限)
  - 水平中央寄り screen_w × 20% ≤ x ≤ screen_w × 80%

採用: 候補のうち信頼スコアが最高のテキスト
```

**根拠:**
iOS ナビゲーションバーのタイトルは常に画面上部かつ水平中央に配置される。
戻るボタン (左端 x≈8%) や右端ボタン (x≈92%) は自動的に除外される。

### 画面サイズの動的推定

```python
screen_h = int(max(OCR bounding-box の y 座標) × 1.05)
screen_w = int(max(OCR bounding-box の x 座標) × 1.05)
```

デバイス情報や画像ファイルを参照せず、OCR 結果のみから画面サイズを推定することで
任意の解像度・プラットフォームに対応する。

---

## 却下した代替案

| 案 | 却下理由 |
|----|---------|
| phash による重複判定で間接的にタイトルを回避 | iOS 白系 UI では phash 距離が 0〜2 で異なる画面を識別不可 |
| Gemini/LLM でタイトルを推定 | レイテンシとコストがリアルタイムクロールに不適 |
| XPath / AccessibilityID で直接取得 | ゲームの多くは AccessibilityID を設定しておらず汎用性がない |
| 解像度別の座標テーブル管理 | デバイス増加に比例して保守コストが増大する |

## Actionable Elements の抽出戦略 (Step 2 — 2026-03-03 追加)

### コンテキスト

Phase 3 の `_find_tappable_items()` はシェブロン (`>`) の絶対 x 座標 (`x > 900`)
やピクセル固定の y 範囲 (`200–1800`) でタップ候補を絞り込んでいた。

問題:
1. シェブロンが右端に出ない UI (ゲーム独自ボタン、テキストリンク) を完全に見落とす
2. 「リセット」「転送」のような **アクションボタンはシェブロンを持たない** ため常に除外
3. 解像度変化でフィルタが機能不全になる

### 決定 — 4-Path 分類アルゴリズム

**座標境界はすべて OCR bounding-box から動的に推定した相対比率で定義する。**

```
STATUS_TOP  = screen_h × 5%   (ステータスバー除外下限)
NAV_BOTTOM  = screen_h × 15%  (nav-bar 領域の下端)
FOOTER_TOP  = screen_h × 80%  (フッターボタン領域の開始)
RIGHT_EDGE  = screen_w × 85%  (シェブロン・アイコン除外右端)
LEFT_MAX    = screen_w × 80%  (コンテンツ列の右端)
LARGE_FONT  = screen_h × 3%   (タイトル相当フォント高さ、Path C/D で除外)
CHEVRON_R   = screen_h × 4%   (シェブロン行の縦判定マージン)
DESC_X_MIN  = screen_w × 40%  (右寄り長文説明テキスト除外)
```

| Path | 条件 | 対象例 |
|------|------|--------|
| A — キーワード | `_ACTION_KEYWORDS` に部分一致 | リセット, 転送, Done, Save |
| B — フッター | `y > FOOTER_TOP` | 画面下部の確認ボタン群 |
| C — シェブロン行 | `RIGHT_EDGE` より右にある `>` / `›` と `y` が `±CHEVRON_R` 以内 | 設定リスト項目 |
| D — コンテンツ fallback | シェブロン数 < 5 かつ nav-bar 下〜フッター上、小フォント | ゲーム独自メニュー |

**除外ルール (共通):**
- `_TAPPABLE_SKIP` に含まれるテキスト (`戻る`, `>`, `›` 等)
- `cy < STATUS_TOP` または `cx > RIGHT_EDGE` (枠外)
- 文字数 0 または 1 文字 (アイコン文字 `_ICON_CHARS` を除く)

### 根拠

`_ACTION_KEYWORDS` を Path A として最優先することで、シェブロンを持たない
「破壊的アクション」(リセット・削除) が確実にリストに載る。
iOS 設定アプリ「一般」画面での実測:
- シェブロン行 (Path C): 9 件 (VPN, BGアプリ更新 等)
- フッターボタン (Path B): 2 件 (リセット, 転送)
- キーワード (Path A): 0 件追加 (フッターと重複のため `seen` set で自動除外)

### 却下した代替案

| 案 | 却下理由 |
|----|---------|
| シェブロン数 >= N で fallback 切替 | N の最適値がアプリ依存；ゲームでは常に 0 になる可能性 |
| Gemini で「タップ可能か」を判定 | レイテンシ・コストがリアルタイムクロールに不適 |
| XPath `//XCUIElementTypeButton` | ゲームの多くは AccessibilityID を設定しておらず汎用性がない |

## 動的 UI を考慮した Settling Wait (Step 3 — 2026-03-03 追加)

### コンテキスト

Phase 3 のクローラーは `wait_after_tap = 3.0` 秒の固定待機でタップ後の描画完了を保証していた。

問題:
1. **静止系画面 (設定アプリ等) では 3 秒の大半が無駄**: 実測では遷移完了まで 0.5〜1.0 秒で十分。
2. **ゲームの Live2D / ローディングエフェクトでは 3 秒で不足**: キャラクターアニメが
   常時動き続ける画面では、固定待機では「静止した」タイミングを保証できない。

### 決定 — phash Settling Wait

**タップ直後の固定待機を廃止し、phash ハミング距離によるアダプティブ静止検知に移行する。**

```
アルゴリズム:
  interval = 0.5s ごとにスクリーンショットを取得し前フレームと比較
  if ハミング距離 ≤ 5:
      → 「静止した」とみなして即座に次処理へ進む
  if 経過時間 ≥ 3.0s:
      → タイムアウト。静止未確認でも強制続行する
```

**閾値 5 の根拠:**

| シナリオ | 典型的な phash 距離 | 判定 |
|---------|---------------------|------|
| 完全同一フレーム | 0 | ✅ 静止 |
| 遷移完了後・静止画面 | 1〜5 | ✅ 静止 |
| iOS 設定画面間 (白系) | 2〜4 | ✅ 静止 (過検知)※ |
| スクロールアニメ中 | 8〜20 | ❌ 動的 |
| Live2D キャラクター | 15〜40 | ❌ 動的 |

※ 白系画面は元々 phash 距離が近いが、Settling Wait においては「静止している」と
判断して問題ない（重複判定とは異なり、静止=OK の文脈）。

タイムアウト 3.0s は Phase 3 の `wait_after_tap` デフォルト値と同じ上限を維持することで
後退互換性を確保しつつ、早期 return により平均待機時間の短縮を実現する。

### 却下した代替案

| 案 | 却下理由 |
|----|---------|
| 固定 wait_after_tap のまま増加 (5s 等) | ゲームによって最適値が大きく異なり汎用性なし |
| Appium の `wait_for_condition` | ゲームは accessibility tree を持たず利用不可 |
| 差分ピクセル数カウント | ゲームの常時アニメで偽陽性が多発。phash の方がノイズ耐性が高い |

### 影響範囲 (Step 3)

- `crawler/lc/driver.py` — `wait_until_stable()` メソッド追加
- `crawler/lc/crawler.py` — `_crawl_impl` 内の `driver.wait(wait_after_tap)` を
  `driver.wait_until_stable()` に置き換え
- `CrawlerConfig.wait_after_tap` フィールドは `wait_until_stable` の内部 timeout と
  独立して残存 (back() 後の待機など他箇所で引き続き使用可能)

## 今後の展望

本 ADR で確立した「相対比率 + 視覚的特徴 + キーワード分類 + アダプティブ静止検知」は
以下の拡張の基盤となる:

- `_screen_fingerprint()` の相対座標化
- ゲーム UI (縦・横画面切替、カスタムフォント) への適応
- 機械学習モデルへの段階的移行 (特徴量として bounding-box 比率を入力)
